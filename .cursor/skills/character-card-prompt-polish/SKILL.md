# 角色卡提示词优化

## 何时启用

用户给出零散设定、草稿或简短人设，需要整理成**可直接用于对话/绘图/ bot 系统提示**的完整角色卡，并强调**形象丰满**与**人物可爱（讨喜、有温度）**。

## 工作流

1. **提取**：列出用户已提供的核心信息（姓名、年龄感、外观、性格、口癖、关系、世界观等）；标注明显空缺。
2. **补全（丰满）**：在不与用户设定矛盾的前提下，用「可推翻的合理默认值」补足：动机、日常习惯、反差萌点、小弱点、社交边界、记忆锚点（标志性事件）。每项都要**可演绎**（能引出具体对话/行为），避免空泛形容词堆叠。
3. **可爱化（讨喜）**：在整体基调允许时，强化**温暖、笨拙、真诚、小小任性或固执**等可亲特质；用具体小动作与口语化反应代替「她很可爱」这类陈述。若用户明确要求冷感/反派/沉重基调，则**不强行卖萌**，只保留「立体、有记忆点」。
4. **结构化输出**：按下方模板输出最终提示词（或中英双语段落，若用户指定语言）。
5. **自检**：是否出现自相矛盾；是否有过长的禁止列表或元说明喧宾夺主；敏感内容遵用户与平台边界。

## 丰满度的着力点

- **外在**：体态与标志性穿搭/配饰、肤色发质、表情习惯、站姿走姿等可画面化的细节（控制长度，挑最有记忆的 3–6 条）。
- **内在**：价值观一句、害怕失去的东西、开心时的样子、生气时的样子（各用**行为**写，不用标签）。
- **关系**：与 1–2 个「他者」的相处模式（朋友/家人/宠物/偶像），便于多轮剧情。
- **语言**：称呼习惯、句长、口头禅或语气词、是否多用比喻/省略——保证**示例台词** 2–4 句。

## 可爱化原则（默认可爱向）

- 用**具体**替代**抽象**：「紧张时会捏袖口」优于「容易害羞」。
- **反差**：外表与反应、强项与软肋的小反差（不 OOC）。
- **温度**：关心人的方式要落地（递水、碎碎念、生硬的安慰），避免油腻或霸总式台词。
- **节制**：可爱来自细节与分寸感，避免堆砌幼态词令或过度卖萌破坏角色完整性。

## 输出模板（按需删减区块）

将最终稿放在单个代码块或明确标题下，便于用户复制。

```markdown
## 角色概要
- 姓名 / 称呼：
- 一句话定位（角色在故事里的功能或气质）：

## 外观与气质
（3–6 条具体可视细节 + 整体印象）

## 性格与行为模式
（动机、习惯、压力反应、与他人边界）

## 语言与示例
- 口癖与句式：
- 示例台词：
  1.
  2.
  3.

## 小传 / 记忆锚点
（可短，但必须能支撑后续展开）

## 演绎提示（给模型或作者）
- 优先强调的特质：
- 建议避免的 OOC：
```

## 交付说明

- 若用户未指定用途，默认产出一版**中文**完整角色卡；若用于英文模型，在文末附**简短 English summary**（可不逐句翻译全文）。
- 保留用户明确要求的关键词与禁忌；新增内容用「推测/可改」类轻微提示标出（除非用户要求不要标注）。
- 用户只要「扩写某一段」，则只输出该段及其依赖的衔接句，避免重写全文。
